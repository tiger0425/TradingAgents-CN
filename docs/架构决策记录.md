## 背景

TradingAgents-cn 是一个基于 LangGraph 的多 Agent 金融分析框架（A股），当前存在以下结构性局限：

1. **固定分析管道** — `propagate("600519", "2026-05-13")` 始终运行同一条 Agent 序列，无法根据客户意图动态调整
2. **无组织概念** — Agent 之间无汇报关系，无角色分工，每次调用是孤立的
3. **无调度机制** — 所有运行依赖手动触发 CLI 命令
4. **无成本控制** — LLM API 调用无预算上限
5. **基础学习能力** — `reflect_and_remember()` 仅做简单文本反思，无结构化技能提炼
6. **知识碎片化** — 分析结果分散在多个被动存储中，Agent 无法主动检索

目标是将其改造为一个**类真实券商研究所的数字金融团队**——有组织架构、有工作节奏、能按客户需求动态响应、能从执行结果中持续学习。

为此，对外部三个热门开源项目（OpenClaw, Hermes Agent, Paperclip）进行了深度对比分析，以确定哪（几）个适合配合使用。

## ADR-1: 三个外部项目的定位与适配结论

### 分析对象

| 项目 | 仓库 | Stars | 语言 | 核心定位 |
|---|---|---|---|---|
| OpenClaw | openclaw/openclaw | ~359K | Node.js/TS | 多平台 AI 个人助手网关 |
| Hermes Agent | NousResearch/hermes-agent | ~120K | Python | 自学型 AI Agent 运行时 |
| Paperclip | paperclipai/paperclip | ~62K | Node.js/TS+React | Agent 编排控制平面 |

### 对比维度

| 维度 | OpenClaw | Hermes Agent | Paperclip |
|---|---|---|---|
| **消息平台覆盖** | 25+ (业界最广) | 14+ | Dashboard 模式（非消息平台） |
| **自学循环** | 静态技能 | GEPA 技能生成 (ICLR 2026 Oral) | N/A（编排层，不运行 AI） |
| **记忆系统** | 扁平 Markdown | 三层（prompt memory + SQLite FTS5 + 8 external providers） | 公司级组织记忆 |
| **预算控制** | 无 | 无 | 原子级预算硬停止 |
| **组织架构** | 单 Agent | 单 Agent | 原生 Org Chart + 角色 + 汇报线 |
| **Heartbeat 调度** | Cron | Cron | 原生 Heartbeat 引擎 + 任务队列 |
| **多公司/多策略隔离** | 无 | 无 | 原生支持（一次部署无限公司） |
| **审计日志** | 无 | SQLite + JSONL session logs | Append-only 不可篡改 |
| **MCP 支持** | 原生深度集成 | Client-side wrapper | 可通过 Adapter 接入 |
| **技术栈一致性** | Node.js（与 TradingAgents Python 不同） | **Python（与 TradingAgents 一致）** | Node.js（不同，但通过 Adapter 解耦） |
| **成熟度** | 最高（最早发布） | 爆发增长中（2026.2 发布） | 最新（2026.3 发布，仅 2 个月） |

### 核心发现：三个项目不是竞争关系，而是并存于不同抽象层次

```
抽象层次：
  Level 3  — 客户交互层     → OpenClaw（消息接收与推送）
  Level 2  — 组织治理层     → Paperclip（任务分派、调度、预算、审批、审计）
  Level 1  — 领域分析层     → TradingAgents（金融多 Agent 分析引擎）
  Level 0  — 持续学习层     → GEPA 方法论（在 TradingAgents 内部实现）
```

**三个项目各负责不同层，不相互替代。**

### 决策

**采用三层组合架构**：Paperclip（组织治理）+ OpenClaw（客户接入）+ TradingAgents（分析引擎）+ GEPA 方法论（学习循环）。

每个项目在最终架构中的角色：

- **Paperclip** → 公司办公室主任+财务总监：组织架构、任务分派、定时调度（晨会/午评/收盘复盘/周日选股）、预算硬停止、Board 审批流、不可篡改审计日志
- **OpenClaw** → 企业微信/电话系统：客户通过微信/Telegram/WhatsApp/Discord 等 25+ 渠道与团队交互，定时推送通知
- **TradingAgents** → 研究所全部研究员：Intent Router（新增）+ 多模式分析管道（改造）+ Agent 辩论 + 数据工具 + 报告生成
- **GEPA 方法论** → 培训发展体系：T+N 复盘 → 评估 → 提取技能文档 → 渐进加载。方法论来自 Hermes（MIT 许可），实现在 TradingAgents 内部

## ADR-2: 为什么 Paperclip 是"最缺的拼图"而非 "nice-to-have"

### 背景

在确定三层架构前，一个关键问题是：TradingAgents 当前最大的结构性缺口是什么？是"缺更多消息渠道"（OpenClaw）还是"缺自学能力"（Hermes）？

### 分析

TradingAgents 当前最致命的五个缺失（按影响排序）：

| 排名 | 缺失能力 | 影响 | 谁解决 |
|---|---|---|---|
| 1 | **定时调度** — 系统不会自己按时运行 | 每天手动 `tradingagents daily`，不可规模化 | Paperclip Heartbeat |
| 2 | **成本控制** — LLM API 费用无上限 | 一次分析调用 4 个 LLM × 多轮辩论 × 日跑 = 月费不可预测 | Paperclip Budget |
| 3 | **组织管理** — 无法同时跑多个策略 | 趋势策略、反转策略、ETF 轮动不能并行 | Paperclip Org+多公司 |
| 4 | **审计追踪** — 决策无法追溯审批 | 出了错不知道哪个 Agent 在哪个环节出了问题 | Paperclip Audit |
| 5 | **多渠道分发** — 只有 CLI 输出 | 用户必须在终端前看结果 | OpenClaw |

自学能力（GEPA）排在第六——它很重要，但比"系统不会自己跑"优先级低。

### 决策

**Paperclip 作为第一优先级集成**，因为它填补的缺口最大且最基础（系统不会自己运行，一切后续能力都建立在这个基础上）。

OpenClaw 作为第二优先级（渠道扩展），GEPA 作为第三优先级（学习优化）。

## ADR-3: "Paperclip 管不了内部 Agent 编排"的边界澄清

### 背景

用户提出了一个关键问题：客户说"我被套了怎么办"，这需要 TradingAgents 内部的 Agent 动态协作来分析——Paperclip 能管这个吗？

### 分析

Paperclip 的管辖范围是**外层编排**（Outer Loop）：

- ✅ "创建一个任务：600519 保本方案"
- ✅ "分配给个股分析组"
- ✅ "控制这项分析的预算 ≤ $2.50"
- ✅ "记录分析结果到审计日志"

Paperclip 的管辖边界（Inner Loop）：

- ❌ 不能动态决定 TradingAgents 内部哪个 Agent（Market Analyst vs Fundamentals Analyst）参与
- ❌ 不能切换 LangGraph 的管道拓扑
- ❌ 不能干预 Agent 间的辩论顺序

### 决策

**双层编排模型**：

```
Paperclip（Outer Loop）       → WHAT + WHO + WHEN + BUDGET
  任务创建 → 部门分派 → 预算控制 → 审计

TradingAgents（Inner Loop）   → HOW
  Intent Router → 管道选择 → Agent 协作 → 报告生成
```

**Inner Loop 必须由 TradingAgents 自己解决**——新增 Intent Router 组件（PRD §6.3），这是改造的最高优先级。

## ADR-4: 知识库五层架构的设计依据

### 背景

用户识别出 PRD 初版缺少知识库章节。当前项目有 5 个被动存储机制（DataCache / AnalysisArchive / TradingMemoryLog / WikiGenerator / MCP Server），但存在严重缺陷：

- Agent 分析时**无法主动查询**任何历史信息
- 只能被动接收 ContextAssembler 在分析开始时注入的固定上下文
- 无执行轨迹（不知道 Agent 在哪个环节做了什么决策）
- 无知识图谱（ticker↔industry↔macro↔decision 的关系未建模）
- 无技能存储（GEPA 学习循环没有输出目标）

### 分析

对比 Hermes Agent 的三层记忆架构和 Paperclip 的文件管理，设计知识库时需要满足：

1. **所有知识完全本地化** — 零外部服务依赖，与 TradingAgents 同进程
2. **Agent 可主动检索** — 不仅是"自动注入"，Agent 在分析管道中可随时调用 `kb.query()`
3. **支持多种查询模式** — 精确（ticker/date）、语义（"找相似"）、关系（图谱遍历）、混合
4. **GEPA 学习循环的 I/O** — 执行轨迹是输入，技能文档是输出
5. **渐进存储成本** — 新知识加入时自动索引，老化知识自动降级（STALE）

### 决策

采用**五层架构**：

```
Layer 4  技能文档库 (Skill Store)        ← GEPA 输出，渐进加载（摘要常驻 ~3k tokens）
Layer 3  知识图谱 (Knowledge Graph)      ← 实体关系连接，JSON 文件 + 内存图遍历
Layer 2  执行轨迹 (Execution Trace)      ← LangGraph callback 自动记录 JSONL
Layer 1  分析存档 (Analysis Archive)     ← 已有，强化 FTS5 全文搜索
Layer 0  原始数据缓存 (DataCache)         ← 已有
+ 语义检索引擎 (SQLite-vec / Chroma)
+ 统一查询接口 (kb.query / kb.similar / kb.related / kb.trace)
```

**技术选型原则**：

| 组件 | 选择 | 理由 |
|---|---|---|
| 向量存储 | SQLite-vec 或 Chroma | 嵌入式，pip install，零运维 |
| Embedding 模型 | BGE-small-zh (~100MB) | 中文优化，本地运行，免费 |
| 知识图谱 | 纯 JSON + 内存遍历 | 规模小（预计 < 10万节点），无需图数据库 |
| 执行追踪 | JSONL 文件 | 追加写入，流式读取，无需额外依赖 |
| 全文搜索 | SQLite FTS5 (Python 内置) | 已有 sqlite3，零额外依赖 |

## ADR-5: 为什么不直接用 Hermes Agent 替代 TradingAgents

### 背景

Hermes Agent 与 TradingAgents 同为 Python Agent 框架，理论上可以直接替代。为什么不？

### 分析

| 维度 | TradingAgents 有 | Hermes Agent 有 |
|---|---|---|
| 金融数据源 | AkShare / 国信证券 / Yahoo Finance / Alpha Vantage（4 个，含 A 股专属） | 无领域数据源 |
| A 股特殊规则 | 涨跌停限制、T+1 交割、交易日历 | 无 |
| 金融 Agent 角色 | Market/News/Fundamentals/Social Analyst + Bull/Bear Researcher + Trader + 3 Risk Debaters + PM | 通用 Agent |
| 多 Agent 辩论机制 | LangGraph 原生的 Bull/Bear 多轮辩论、Aggressive/Conservative/Neutral 三方风险辩论 | 单 Agent 循环 |
| 金融专业输出 | 买入/卖出建议 + 评级 + 置信度 + 回测 | 通用任务输出 |

**TradingAgents 的壁垒是领域知识深度**，Hermes 的壁垒是通用自学能力。两者不是替代关系。

### 决策

**保留 TradingAgents 作为核心分析引擎，从 Hermes 吸收 GEPA 方法论**（MIT 许可，方法论层面可自由实现），在 TradingAgents 内部构建学习循环。不替换 LangGraph 架构。

## ADR-6: 数据存储分层

### 背景

Paperclip（PostgreSQL）和 TradingAgents（文件系统）的存储需要统一规划。

### 决策

```
Paperclip 的 PostgreSQL:
  存储: 组织架构、Agent 配置、任务/Issue、预算/费用、审计日志、审批记录
  理由: 需要 ACID（预算硬停止不能出错），需要关系查询（Agent ↔ 任务 ↔ 费用）

TradingAgents 的文件系统 (~/.tradingagents/):
  存储: 分析结果、数据缓存、执行轨迹、知识图谱、技能文档、向量嵌入
  理由: 分析结果天然适合 Markdown/JSON 文件（人类可读、可 git 版本控制、可直接编辑）
       向量嵌入和知识图谱规模小，嵌入式方案比外部服务更简单
       与 Paperclip 解耦：即使 Paperclip 不可用，TradingAgents 仍可独立运行
```

**两个存储层之间不直接通信。** Paperclip 通过 Adapter 调用 TradingAgents CLI，TradingAgents 通过 Paperclip API 回传任务状态和费用数据。

## ADR-7: 实施优先级

### 决策

```
P0（必须有） — Phase 1-2（10 周）
  ① Paperclip 部署 + 组织架构 + 晨会/收盘 Heartbeat
  ② Intent Router（TradingAgents 内部新增最高优先级组件）
  ③ BreakevenPipeline + StandardPipeline 改造
  ④ OpenClaw 多平台接入

P1（应该有） — Phase 3（3 周）
  ⑤ 预算控制 + Board 审批
  ⑥ WeeklyScreeningPipeline + MarketReviewPipeline

P2（会有） — Phase 4-5（持续）
  ⑦ 知识库 Layer 2-4（执行轨迹 + 知识图谱 + 技能文档）
  ⑧ GEPA 学习循环（T+N 复盘 + 技能提炼）
  ⑨ 语义检索引擎
  ⑩ 行业研究组扩展（消费 → 科技 → 金融）
```

**先让团队跑起来（P0），再让团队管好钱（P1），最后让团队越用越聪明（P2）。**
