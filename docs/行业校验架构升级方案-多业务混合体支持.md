## 一、触发事件回顾

600418 江淮汽车本次分析触发 `IndustryVerifier` 报告如下：

```
IndustryVerifier: consistency check failed for 600418:
1. 报告日期存在严重矛盾，市场分析日期为2025-02-20，新闻分析日期为2025-03-17，
   而基本面分析日期为2026年6月4日，数据时间线混乱
2. 基本面分析中大量提及华为尊界品牌、蔚来代工等乘用车业务，但报告行业为商用载货车，
```

这 4 条 issue 表面是孤立 bug，实际是**同一架构假设被多业务混合体打穿**的系统性问题。

## 二、问题诊断

### 2.1 问题 #1：时间线混乱（🔴 高 — 数据可信度受损）

#### 现象
3 个 Analyst 写出 3 个不同 `analysis_date`：

| Analyst | analysis_date | 推断来源 |
|---|---|---|
| Market Analyst | **2025-02-20** | OHLCV 数据最后一行 |
| News Analyst | **2025-03-17** | 新闻最新发布日期 |
| Fundamentals Analyst | **2026-06-04** | system prompt 中的 `current_date` |

#### 根因（2 个串联缺陷）

**缺陷 A — 第二轮结构化调用未传 `current_date`**

`tradingagents/agents/analysts/{market,fundamentals,news,social_media}_analyst.py` 全部遵循以下模式：

```python
# 第一轮 LLM 调用（带工具）— 注入 current_date ✅
prompt = ChatPromptTemplate.from_messages([
    ("system", "{system_message}

For your reference, the current date is {current_date}."),
    MessagesPlaceholder(variable_name="messages"),
])
prompt = prompt.partial(current_date=state["trade_date"])

# 第二轮 LLM 调用（结构化输出 / 填 Pydantic schema）— 缺失 current_date ❌
fmt_prompt = (
    f"Reformat the following technical analysis for {company_name} ...

"
    f"Analysis:
{content}"
)
# ↑↑ 这里没有任何 current_date 上下文 ↑↑
content = invoke_structured_or_freetext(structured_llm, llm, fmt_prompt, ...)
```

LLM 在第二轮只能从 raw analysis content 中"猜"日期。Market/News 看到的数据自带日期 → 沿用数据日期；Fundamentals 看到财务三表无"最后交易日"概念 → 回退到 system prompt 日期。3 个 Analyst 行为分叉。

**缺陷 B — 数据日期 ≠ 分析日期未区分**

| 概念 | 来源 | 当前是否区分 |
|---|---|:---:|
| 分析日期（trade_date） | `executor.py:222` `date.today().isoformat()` = `"2026-06-04"` | ✅ 存在 |
| 数据日期（as_of_data_date） | mootdx OHLCV 最后一行 = `"2025-02-20"` | ❌ 缺失 |

LLM 在第二轮看到 OHLCV 表里的 `Date` 列，自然把数据日期当成 analysis_date。系统从未在 prompt 里告诉 LLM："工具返回的 Date 是数据日期，不是分析日期"。

#### 修复方向

| 方案 | 改动量 | 优先级 |
|---|---|---|
| 第二轮 `fmt_prompt` 末尾追加 `

The analysis_date MUST be {current_date}. You MUST use {current_date} as the analysis_date, not any date found in tool data.` | 4 个 analyst × 5 行 | P0 |
| `agent_states.py` 加 `as_of_data_date: str` 字段；数据工具返回时回填 | 数据层 + state 字段 | P0 |
| Pydantic schema 把 `analysis_date` 改为 `Literal[trade_date]` 或加 `field_validator` | 1 schema | P1 |
| 注入层显式区分"数据日期"与"分析日期"语义 | injection_contract | P2 |

### 2.2 问题 #2：行业配对不支持多行业（🟡 中 — 焦点偏移）

#### 现状：单值假设贯穿 4 层

```
tradingagents/dataflows/a_stock_data.py:get_industry()
    │ 三层 fallback：mootdx F10 → 东方财富 f127 → F10 正则
    ▼
返回 1 个字符串: "商用载货车"
    │
    ▼
tradingagents/industry/classifier.py:IndustryClassifier.classify()
    │ _split_primary_secondary() 仅拆分罗马数字后缀
    ▼
IndustryResult(primary="商用载货车", secondary="")
    │
    ▼
tradingagents/industry/frameworks.py:IndustryFramework.lookup()
    │ 4 步模糊匹配（精确 keyword → 子串 → input in keyword → token 匹配）
    ▼
返回 1 个 framework（automotive）
    │
    ▼
tradingagents/industry/verifier.py:verify_industry_consistency(industry, report)
    │ 拿 1 个 framework 的 anti_patterns 去扫
    ▼
    单维度校验
```

每一层都把"一家公司 = 一个行业"作为隐式约束。**没有主+副并存的语义，没有业务段比例的概念。**

#### 江淮汽车的"多业务"困境

| 业务 | 收入占比（估算） | 系统是否感知 |
|---|:---:|:---:|
| 商用载货车 | ~55% | ✅ primary |
| 乘用车（含尊界/蔚来代工） | ~30% | ❌ 完全忽略 |
| 新能源车 | ~15% | ❌ 完全忽略 |

基本面 LLM 在生成报告时被"尊界/蔚来代工"热点吸引，写出大段乘用车内容 — 因为**乘用车在 automotive 框架里被视作同行业合法表述**，anti_patterns 不会拦截。

#### 修复方向

**核心思路：`IndustryResult` 扩展为多业务段结构**

```python
@dataclass
class BusinessSegment:
    industry: str           # "商用载货车"
    framework_key: str      # "automotive"
    revenue_share: float    # 0.55
    source: str             # "财报" | "公告" | "概念板块"
    is_primary: bool

@dataclass
class IndustryResult:
    ticker: str
    primary_industry: str                  # segments[0].industry（向后兼容）
    confidence: float
    business_segments: list[BusinessSegment]  # 新增
    source: str
```

**`IndustryClassifier` 升级路径**：

```
数据源                                提取方法
─────────────────                  ──────────────
财报"主营业务分行业"表              a_stock_data.get_financial_statements() 解析
akshare 概念板块                     stock_board_industry_cons_em
公告扫描（30天）                     announcements 提及"尊界"/"蔚来"/"换电" → 加权
─────────────────────────────────── ──────────────────────────────────────
                                      ↓
                              合并为 BusinessSegment 列表
                              按 revenue_share 降序
                              share < 0.05 的丢弃
```

### 2.3 问题 #3：幻觉防护有盲区（🔴 高 — 报告质量受损）

#### 现有 3 类幻觉检测的盲区

| 检测器 | 实现位置 | 机制 | 盲区 |
|---|---|---|---|
| `verify_metric_sources` | `verifier.py:110-177` | 数字是否在工具返回数据中出现（含百分比↔小数自动转换） | **只查"数字是否存在"，不查"数字描述的业务是否在主行业范围"** |
| `detect_foreign_terms` | `verifier.py:181-239` | 扫描 EPA/Class 8/NYSE/FOMC 等海外术语 | **只查"是不是境外市场"，不查"是不是本行业的对应市场"** |
| `verify_industry_consistency` (LLM 层) | `verifier.py:317-355` | 截取 2000 字符 + LLM 判断行业匹配 | **只查"主行业单值"，不查"业务比例 / 焦点漂移"** |

600418 报告实际触发的是**第 3 个**（`method="llm"`）—— 规则层的 `automotive` 框架 anti_patterns **只有 12 个 SaaS 反模式**（续约率/LTV/CAC/ARR/MAU 等），完全不包含"乘用车偏移"反模式。

#### 根因再深入：anti_patterns 的"硬编码"结构无法覆盖跨业务偏移

`tradingagents/industry/config/industry_frameworks.json` 的 `automotive` 框架当前结构：

```json
"automotive": {
  "name": "汽车与商用车",
  "keywords": ["汽车", "商用车", "整车", "乘用车", "汽车制造", "商用载货车"],
  "correct_metrics": ["月度销量", "产能利用率", "毛利率", "新能源渗透率"],
  "anti_patterns": [
    "续约率", "LTV/CAC", "ACV", "ARR", "NRR",
    "TOP100客户", "云端订阅", "月活跃用户",
    "客户获取成本", "用户留存率", "净推荐值NPS", "MAU"
  ],
  "peer_companies": ["上汽集团", "比亚迪", "江淮汽车", "福田汽车"],
  "context_instruction": "该企业属于汽车/商用车行业..."
}
```

**三个核心缺陷**：

1. ❌ **无 `industry_type` 字段** — `_type_rules["manufacturing"]` 的通用反模式（光模块ASP、GPU需求、CPO技术路线等）只在 LLM auto-gen 框架时进 prompt，**对已有 automotive 框架完全无效**
2. ❌ **`_with_injection()` 只做 truncation，不做规则合并**（`frameworks.py:100-106`）— `_type_rules` 与具体框架的 anti_patterns 是分离的两套数据，互不引用
3. ❌ **automotive 框架 `keywords` 列表包含"乘用车"** — 这导致乘用车内容（华为尊界、蔚来代工）在规则层**不被视为反模式**。`IndustryFramework.lookup("商用载货车")` 命中 automotive 框架后，LLM 生成的报告中出现"乘用车"内容时，关键词匹配层完全不会拦截——因为"乘用车"是 automotive 框架 keywords 里的**合法术语**。规则层 anti_patterns 只拦截 SaaS 指标（续约率/LTV/CAC），乘用车偏移只能通过 LLM 语义层（`_llm_check`）被动发现，且需要在 prompt 中明确告诉 LLM："虽然该公司有乘用车业务，但当前分析以商用载货车为主行业"

#### 补充：debate 层的行业约束盲区

`bull_researcher` / `bear_researcher` 辩论 Agent 虽然接收了 anti_patterns 注入（`context_manager.py:inject_context()`），但 **debate topic 的选择不受行业框架约束**。600418 报告的 Investment Plan（第 80-87 行）显示 debate 围绕"换电重卡"展开多空辩论，但 market/fundamentals/news 三份前置报告均未提供换电重卡数据——这是 **"debate 主题漂移"** 问题：debate Agent 自选了一个当前分析无法支撑的辩论主题。修复方向：在 debate 节点的 system prompt 中注入 `business_segments`，要求辩论必须围绕**主行业**的核心变量展开，副行业仅作为"风险因素"提及。

#### 修复方向

**3 层幻觉防护升级**：

| 层级 | 改造 | 解决什么 |
|---|---|---|
| **L0 数字+上下文** | `verify_metric_sources` 升级为 `verify_metric_with_context`：把数字周围 30 字上下文喂给 LLM，判断"这个数字+上下文是否在工具数据中 + 描述的业务是否在主行业范围" | 数字伪造 + 业务错位 |
| **L1 反模式+类型继承** | `industry_frameworks.json` 的 `automotive` 加 `"industry_type": "manufacturing"`；`_with_injection()` 在 truncation 前先做 `_type_rules[industry_type].anti_patterns ∪ framework.anti_patterns` 并集 | 跨类型偏移（汽车讲 GPU 需求） |
| **L2 焦点漂移检测** | 新增 `verify_topic_focus(report, business_segments)`：用 LLM 对报告做主题分类（"商用车销量"/"乘用车合作"/"财务结构"），算每主题 token 占比，与 segments 的 `revenue_share` 做 KL 散度 | "整篇讲乘用车" vs "商用车收入只占 10%" |

### 2.4 问题 #4：多业务混合体无法分清主业务（🟡 中 — 焦点漂移无法量化）

#### 现状：完全没有"业务段"概念

```python
# 现在的 600418 行业链路
get_industry("600418") 
  → mootdx F10 返回 "商用载货车"          # 按 F10 行业代码单值
  → 强行框进 automotive 框架               # 没有"还有乘用车"概念
  → 反模式只查 SaaS 词                     # 完全没考虑业务偏移
```

**没有任何代码**做以下事情：
- ❌ 从财报"分行业收入"表提取各业务收入占比
- ❌ 从公告/概念板块识别"该公司还有 X 业务"
- ❌ 报告中按业务占比分配指标权重
- ❌ 校验"乘用车段落篇幅"是否与"乘用车收入占比"匹配

#### 修复方向

**1. 数据源（业务段检测输入）**

```
财报"主营业务分行业"表  ──→  a_stock_data.get_financial_statements() 解析
  例: { "商用载货车": 0.55, "乘用车": 0.30, "新能源": 0.15 }

akshare 概念板块        ──→  stock_board_industry_cons_em
  例: ["华为汽车概念", "新能源车", "换电概念"]

公告扫描（最近 30 天）   ──→  announcements 关键词加权
  例: 提及"尊界"×5, "蔚来"×3, "换电"×2 → 副业务段加权
```

**2. `IndustryResult` 升级（兼容旧 API）**

```python
@dataclass
class IndustryResult:
    ticker: str
    primary_industry: str                  # "商用载货车" — 旧 API 兼容
    confidence: float
    business_segments: list[BusinessSegment]  # 新字段
    # [
    #   BusinessSegment(industry="商用载货车", framework_key="automotive", 
    #                  revenue_share=0.55, source="财报", is_primary=True),
    #   BusinessSegment(industry="乘用车", framework_key="passenger_auto",
```

**3. 报告模板按主业务**

```
报告主标题：江淮汽车（600418）— 商用载货车行业分析
主指标：   月度销量 / 产能利用率 / 新能源渗透率（automotive 主框架）
副段：     业务结构 — 商用车 55% / 乘用车 30% / 新能源 15%
          副段仅展示与主行业"显著不同"的指标，标注 [副业务]
```

**4. 校验按业务比例**

```python
def verify_with_business_mix(segments, report):
    primary, others = segments[0], segments[1:]
    
    # 主行业 anti_patterns 全量扫描
    primary_hits = scan(framework[primary.framework_key], report, strict=True)
    
    # 副行业只查"互斥冲突"（如 SaaS 指标出现在硬件公司）
    secondary_hits = []
    for seg in others:
        secondary_hits += scan(framework[seg.framework_key], report,
                               strict=False, allow_topics=seg.allowed_topics)
    
    # 焦点漂移检测（KL 散度）
    topic_focus = measure_topic_focus(report)   # 报告主题占比
    expected_focus = [s.revenue_share for s in segments]
    drift = kl_divergence(topic_focus, expected_focus)
    
    if drift > THRESHOLD:
        return warn("报告话题占比与业务收入占比严重偏离")
```

## 三、根因耦合关系

```
        ┌─ Q1 时间线（数据日期 vs 分析日期）
        │      ↓
Q2 多行业 ─┤  IndustryResult 单值 → 焦点偏移无法量化
        │      ↓
        └─ Q3 幻觉（焦点漂移 / 跨行业指标）
              │
              └─ Q4 多业务混合体（业务段 + 比例分配）
```

**核心架构缺陷（一句话总结）**：

> `IndustryResult` / `IndustryFramework.lookup()` / `verify_industry_consistency` 三层都是**单值假设**，系统把"一家公司 = 一个行业 = 一份报告"作为隐式约束。江淮汽车的"商用车+乘用车+新能源"三业务结构 + 多源数据时间戳不一致 + LLM 焦点漂移，把这个约束**同时打穿**。

## 四、升级方案

### 4.1 总体设计原则

| 原则 | 实现 |
|---|---|
| **主行业优先** | 报告模板、anti_patterns 严格度、指标权重 都按主行业 → 副行业降级 |
| **业务段透明** | 任何分析开始前先解析 `business_segments`，所有下游节点共享 |
| **焦点可量化** | 引入 KL 散度检测话题占比 vs 业务占比偏离 |
| **数据日期与分析日期分离** | `as_of_data_date` ≠ `analysis_date`，分别在 schema 中固化 |
| **向下兼容** | 旧 API（`IndustryResult.primary`）保留，升级路径渐进 |

### 4.2 详细改造方案

#### 4.2.1 P0 — 时间线统一（Q1 修复）

**改动 1：第二轮结构化调用硬注入 `current_date`**

4 个 analyst 文件（`tradingagents/agents/analysts/{market,fundamentals,news,social_media}_analyst.py`）的 `fmt_prompt` 末尾追加：

```python
fmt_prompt = (
    f"Reformat the following technical analysis for {company_name} "
    f"({state['company_of_interest']}) into a structured market report.

"
    f"Analysis:
{content}

"
    f"IMPORTANT: The current analysis date is {current_date}. "
    f"You MUST set analysis_date to '{current_date}', NOT to any date "
    f"found inside the analysis content. Dates in tool data (OHLCV, news) "
    f"are DATA DATES, not analysis dates."
)
```

**改动 2：Pydantic schema 加 `field_validator`**

```python
# tradingagents/agents/schemas.py
from pydantic import field_validator

class MarketReport(BaseModel):
    analysis_date: str = Field(description="...")
    
    @field_validator("analysis_date")
    @classmethod
    def _validate_analysis_date(cls, v: str, info) -> str:
        expected = info.context.get("current_date") if info.context else None
        if expected and v != expected:
            raise ValueError(
                f"analysis_date must equal current_date '{expected}', got '{v}'"
            )
        return v
```

**改动 3：state 加 `as_of_data_date` 字段**

```python
# tradingagents/agents/utils/agent_states.py
class AgentState(TypedDict):
    trade_date: str                 # 分析日期（不变）
    as_of_data_date: str            # 数据日期（新增）
    # ...
```

#### 4.2.2 P0/P1 — IndustryResult 扩展（Q2+Q4 修复）

**改动 1：`IndustryResult` 加 `business_segments`**

```python
# tradingagents/industry/classifier.py
@dataclass
class BusinessSegment:
    industry: str
    framework_key: str
```

```python
class IndustrySegmentDetector:
    def detect(self, ticker: str) -> list[BusinessSegment]:
        segments = []
        
        # 1. 财报"主营业务分行业"表
        fin = get_financial_statements(ticker)
```

```python
    industry_type = result.get("industry_type")
```

> ⚠️ 以下代码为设计草案。标记 `# TODO` 的辅助函数（`measure_topic_focus`、`kl_divergence`、`_classify_topics`）需在实施时补充完整实现。

```python
# tradingagents/industry/verifier.py（新增方法）
@staticmethod
def verify_topic_focus(
    report: str,
    segments: list[BusinessSegment],
    quick_llm: Any = None,
) -> dict[str, Any]:
    """检测报告话题占比 vs 业务收入占比的 KL 散度。"""
    if not segments or not quick_llm:
        return {"verdict": "pass", "issues": [], "method": "topic_focus"}
    
    # 1. LLM 分类报告各段落的主题
    topic_classification = _classify_topics(quick_llm, segments, report[:3000])
    # 例: {"商用车销量": 0.20, "乘用车合作": 0.55, "财务结构": 0.25}
    
    # 2. 期望分布 = segments 的 revenue_share
    expected = {s.industry: s.revenue_share for s in segments}
    
    # 3. KL 散度
    drift = kl_divergence(topic_classification, expected)
    
    if drift > 0.5:    # 阈值待调
        return {
            "verdict": "warn",
            "issues": [
                f"报告话题占比与业务收入占比严重偏离 (KL={drift:.2f}): "
                f"实际 {topic_classification} vs 期望 {expected}"
            ],
            "method": "topic_focus",
        }
    return {"verdict": "pass", "issues": [], "method": "topic_focus"}
```

```python
def verify_metric_with_context(
    report: str,
    tool_messages: list[dict],
    segments: list[BusinessSegment],
    quick_llm: Any = None,
) -> dict[str, Any]:
    """在原 verify_metric_sources 基础上，加"上下文业务归属"判断。"""
    # 原数字溯源（保留）
    base_result = IndustryVerifier.verify_metric_sources(report, tool_messages)

        return base_result
    
    # 对未找到来源的数字 + 上下文，让 LLM 判断"业务归属"
    issues = base_result.get("issues", [])
    context_issues = []
    for issue in issues:
        # 提取数字 + 周围 30 字上下文
        ctx = _extract_context(issue)
        # LLM 判断这个上下文描述的业务是否在 segments 范围内

        if not belongs:
            context_issues.append(
                f"数字上下文描述的业务不在主行业范围: {ctx}"
            )
    
    return {
        "verdict": "warn" if context_issues else "pass",
        "issues": base_result["issues"] + context_issues,
        "method": "metric_context",
    }
```

当前 `ReportRenderer` 是**三段式骨架**（调用 `render_market_report()`、`render_fundamentals_report()` 等独立函数组装各 Analyst 报告）。改造思路：在组装各段**之前**插入业务结构段，作为报告的顶层概览。

```python
# 在 ReportRenderer 现有 render() 方法中，在各 analyst 报告之前插入
def render(self, final_state, plan=None):
    industry_result: IndustryResult = final_state.get("industry_result")
    segments = industry_result.business_segments if industry_result else []
    
    # 新增：业务结构段（放在所有 analyst 报告之前，作为报告概览）
    biz_section = self._render_business_structure(segments) if len(segments) > 1 else ""
    
    # 原逻辑：组装各 analyst 报告（保持三段式骨架不变）
    parts = []
    for agent in plan.agents:
        report_text = self._render_agent_report(agent, final_state)
        if report_text:
            parts.append(report_text)
    
    report = biz_section + "

" + "

".join(parts)
    
    # 副行业附加段（仅展示与主行业"显著不同"的指标，放在最后标注 [副业务]）
    for seg in segments[1:]:
        diff_metrics = _diff_metrics(seg, segments[0].industry)  # TODO
        if diff_metrics:
            report += f"

## {seg.industry} [副业务]"
            for m in diff_metrics:
                report += f"
- {m}"
    
    return report

def _render_business_structure(self, segments: list[BusinessSegment]) -> str:
    """渲染业务结构概览段。"""
    lines = ["## 业务结构"]
    for seg in segments:
        tag = "[主业务]" if seg.is_primary else "[副业务]"
        lines.append(f"- {tag} {seg.industry}: {seg.revenue_share*100:.0f}% (来源: {seg.source})")
    return "
".join(lines)
```
