## 2. 产品概述

### 2.1 一句话描述

一个按照真实证券公司研究所组织架构和工作节奏运行的、能接收客户定制需求并持续从执行结果中学习的**数字金融团队**。

### 2.2 产品定位

当前 TradingAgents 是一个"给定股票代码+日期，输出买入/卖出建议"的**固定分析管道**。PRD 目标是将它改造成一个**完整的数字券商研究所**：

| 维度 | 当前状态 | 目标状态 |
|---|---|---|
| **组织形态** | 无组织概念，单次调用 | 完整研究所架构：所长/宏观策略组/行业研究组/个股分析组/交易组/风控组/投顾组/学习组 |
| **工作节奏** | 手动触发 `tradingagents batch` | 自动按交易日节律运行：晨会→午评→收盘复盘→次日预案；周末选股推荐 |
| **任务类型** | 只有一种：买卖建议 | 6+ 种分析模式：标准分析/保本方案/周选股/市场复盘/政策影响/客户定制 |
| **客户交互** | CLI 命令行 | 客户通过微信/Telegram/WhatsApp 等 25+ 渠道发送请求，团队接收→分析→回复 |
| **知识库** | 分散的被动存档（文件/内存日志） | 五层统一知识库：原始数据→分析存档→执行轨迹→知识图谱→技能文档。Agent 分析中可主动查询，"我问你答"变成"你自己去查" |
| **学习能力** | 基础 `reflect_and_remember()` | GEPA 闭环：决策→N天后复盘→自动提炼技能文档→下次分析自动加载→越用越准 |

### 2.3 核心理念

> **这不是一个"会生成分析报告的脚本"，而是一个"有组织、有节奏、会学习的券商研究所"。**

## 3. 用户画像

### 3.1 主要用户

| 用户类型 | 典型场景 | 核心需求 |
|---|---|---|
| **个人投资者** | 持有 5-10 只 A 股，每天需要了解持仓动态、收到选股推荐、在亏损时获得解套方案 | 每日晨报 + 持仓预警 + 定制化问题解答 |
| **小型私募操盘手** | 管理 500 万-2000 万组合，需要每周选股、每日风控、盘中调仓建议 | 多策略并行 + 预算控制 + 完整审计 |
| **家族办公室经理** | 跨市场配置（A股+港股+美股），需要多币种、多时区覆盖 | 多公司隔离 + 多市场数据源 + 统一 Dashboard |

### 3.2 用户行为

```
典型一周:

周日 21:00  收到推送: "下周推荐 5 只标的 + 策略周报"
周一 08:50  收到推送: "今日晨报 — 隔夜外盘 + 公告汇总 + 持仓预警"
周一 10:30  微信发消息: "600519 公告怎么看？帮我解读"
周一 12:00  收到推送: "午评 — 上午异动 + 下午策略"
周一 15:10  收到推送: "收盘复盘 — 收益归因 + 明日预案"
周三 14:00  微信发消息: "我被套了5%，怎么解套？"
周五 16:00  收到推送: "本周收益归因 + 经验提炼 + 技能更新"
```

## 4. 团队组织架构

### 4.1 对标原型 — 华创证券研究所真实架构

以中型券商研究所为蓝本，提取核心岗位，去除需要物理在场的职能（如路演、机构销售拜访），保留全部可由 AI Agent 执行的岗位：

```
                        ┌─────────────────────┐
                        │   所长（Director）     │
                        │  战略方向 + 任务分派    │
                        │  重大决策审批 + 质量控制 │
                        └──────────┬──────────┘
                                   │
        ┌──────────────┬───────────┼───────────┬──────────────┐
        │              │           │           │              │
  ┌─────▼─────┐ ┌─────▼─────┐ ┌───▼───┐ ┌────▼────┐ ┌─────▼─────┐
  │宏观策略组  │ │行业研究组  │ │个股组 │ │交易组   │ │投顾服务组  │
  │Macro      │ │Sector     │ │Equity │ │Trading  │ │Advisory   │
  └─────┬─────┘ └─────┬─────┘ └───┬───┘ └────┬────┘ └─────┬─────┘
        │              │           │           │              │
  ┌─────▼─────┐ ┌─────▼─────┐      │           │              │
  │风控合规组  │ │学习发展组  │      │           │              │
  │Risk/Compl │ │Learning   │      │           │              │
  └───────────┘ └───────────┘      │           │              │
                                   │           │              │
                          ┌────────▼───────────▼──────────────▼──┐
                          │           客户（Client）               │
                          │  通过 OpenClaw 25+ 渠道与团队交互       │
                          └──────────────────────────────────────┘
```

### 4.2 各岗位职责与 Agent 配置

#### 所长（Director of Research）— 1 名

| 属性 | 说明 |
|---|---|
| **类比现实岗位** | 研究所所长 / CIO |
| **职责** | 接收客户请求或 Paperclip 定时任务 → 分解为子任务 → 分派给对应部门 → 审核最终输出 → 重大决策提交 Board 审批 |
| **Agent 类型** | 通用 LLM（GPT-5.4 / Claude Opus），不直接调用交易工具 |
| **所属平台** | Paperclip（CEO Agent） |
| **预算** | $20/月（低频，每次任务分解 ~$0.10-0.30） |
| **输出** | 任务分解树、部门指派、质量审核意见 |

#### 宏观策略组（Macro Strategy Team）— 2 名

| 岗位 | 职责 | Agent 类型 |
|---|---|---|
| **宏观研究员** | 跟踪：央行政策（MLF/LPR/降准降息）、财政政策、CPI/PPI/PMI、社融数据、汇率、美债收益率 | TradingAgents Macro Analyst |
| **策略研究员** | 综合宏观+市场数据 → 大盘择时判断（牛/熊/震荡）→ 风格轮动建议（价值/成长/大小盘） | TradingAgents Strategy Analyst |

| 属性 | 说明 |
|---|---|
| **数据源** | 国信证券宏观经济接口、AkShare 宏观模块、人民银行官网、财政部官网 |
| **工作节奏** | 每周日生成策略周报；每个交易日盘前更新宏观环境 |
| **输出** | 宏观环境简报（注入所有下游 Agent）、策略周报、大盘风险提示 |
| **预算** | $15/月（低频高价值，周报+突发政策） |

#### 行业研究组（Sector Research Team）— 按行业拆分

覆盖行业（第一期 3 个）：

| 行业 | 研究员 | 覆盖标的示例 |
|---|---|---|
| 消费 | Sector Analyst - Consumer | 白酒（600519）、食品、家电、免税 |
| 科技 | Sector Analyst - Tech | 半导体、AI、信创、消费电子 |
| 金融 | Sector Analyst - Finance | 银行、保险、券商 |

| 属性 | 说明 |
|---|---|
| **职责** | 每周日扫描所属行业 → 筛选进入上升通道/资金流入的子行业 → 从中筛选 Top 3 个股 → 提交给所长汇总 |
| **数据源** | 行业指数、资金流向、北向资金、机构调研、行业研报 |
| **工作节奏** | 每周日批量；盘中异动事件触发（行业级别） |
| **输出** | 行业周报、子行业轮动图、Top 3 推荐 + 理由 |
| **预算** | $30/月（每周批量，覆盖 3 个行业） |

#### 个股分析组（Equity Research Team）— 4 名

这是当前 TradingAgents 已有的团队，职责不变，但增加多种分析模式。

| 岗位 | 对应代码 | 职责 |
|---|---|---|
| 技术面分析师 | `market_analyst.py` | K线形态、均线系统、MACD/RSI/BOLL、量价关系 |
| 基本面分析师 | `fundamentals_analyst.py` | ROE/毛利率/现金流/估值（PE/PB/PS）、财报分析 |
| 新闻分析师 | `news_analyst.py` | 公告解读、新闻事件影响、行业政策跟踪 |
| 舆情分析师 | `social_media_analyst.py` | 社交情绪、股吧讨论热度、机构调研动向 |

| 属性 | 说明 |
|---|---|
| **下属子团队** | 多方研究员（Bull Researcher）、空方研究员（Bear Researcher）、研究主管（Research Manager） |
| **新增能力** | Intent Router → 根据任务意图选择不同分析管道（标准/保本/政策影响/客户定制） |
| **预算** | $60/月（最高频使用，每日+客户按需） |

#### 交易组（Trading Desk）— 2 名

| 岗位 | 职责 | Agent 类型 |
|---|---|---|
| **交易员** | 接收研究主管的买卖建议 → 拆解为具体交易方案（价格区间/数量/时机）→ 输出可执行指令 | Trader Agent |
| **组合经理** | 综合持仓现状 + 交易方案 + 风险约束 → 最终调仓决策 → 输出完整交易计划 | Portfolio Manager |

| 属性 | 说明 |
|---|---|
| **输出** | 具体买卖价格、数量、仓位比例、止损/止盈触发条件 |
| **风控约束** | 单票上限 20%、行业上限 40%、总仓位上限由客户设定 |

#### 风控合规组（Risk & Compliance）— 3 名

| 岗位 | 对应代码 | 职责 |
|---|---|---|
| 激进风控师 | `aggressive_debator.py` | "风险可控，应该抓住机会" — 从收益最大化角度论证 |
| 保守风控师 | `conservative_debator.py` | "安全第一，不要冒险" — 从最大回撤控制角度论证 |
| 中立风控师 | `neutral_debator.py` | 综合两方观点，提出平衡方案 |

| 属性 | 说明 |
|---|---|
| **输出** | 风险辩论记录 + 风控建议 + 仓位限制硬约束 |
| **合规检查** | 涨跌停限制、T+1 交割规则（A股特有）、单票集中度、行业集中度 |

#### 投顾服务组（Advisory Team）— 1 名

| 属性 | 说明 |
|---|---|
| **类比现实岗位** | 投资顾问 / 客户经理 |
| **职责** | 接收客户消息（通过 OpenClaw）→ 理解客户意图 → 归类为不同需求类型 → 提交给所长分派 → 将团队分析结果以客户友好的语言返回 |
| **Agent 类型** | 通用 LLM + 意图识别 + 客户偏好记忆 |
| **所属平台** | Paperclip（Advisory Agent），通过 OpenClaw 与客户交互 |
| **输出** | 客户友好的回复（非纯学术报告，包含操作建议 + 风险提示） |

#### 学习发展组（Learning & Development）— 1 名

| 属性 | 说明 |
|---|---|
| **类比现实岗位** | 研究所培训主管 / 投研方法论总结 |
| **职责** | T+N 天后拉取历史决策 → 对比预测 vs 实际走势 → 评估正确/错误原因 → 提炼可复用技能文档 → 更新 MEMORY.md → 标记避坑要点 |
| **方法论** | GEPA（Genetic-Pareto Prompt Evolution）— 观察→评估→提取→提炼→技能文档 |
| **Agent 类型** | 轻量 LLM（复盘用，成本极低） |
| **输出** | Skill.md 文档、MEMORY.md 更新、USER.md 偏好更新、错误模式库 |
| **触发条件** | T+5 自动触发（A股决策至少5个交易日后复盘）; 同类任务累计 3 次成功 → 自动生成技能 |

## 5. 工作流程与节奏

### 5.1 完整一周时间表

```
📅 周六（深度研究日）
  10:00  Paperclip Heartbeat → 宏观策略组 + 行业研究组
         ├── 回顾本周全部政策/数据变化
         ├── 行业研究员完成各自行业的深度扫描
         └── 输出: 本周总结 + 下周展望初稿

📅 周日（推荐生成日）
  09:00  Paperclip Heartbeat → 宏观策略组 + 行业研究组 + 所长
         ├── 宏观研究员生成下周宏观环境预判
         ├── 行业研究员生成行业轮动 + Top 3 选股
         ├── 所长汇总 → 策略周报 + 5 只推荐标的
         └── 输出: 推送客户（OpenClaw → WeChat/Telegram）
  21:00  Paperclip Heartbeat → 投顾服务组
         └── 生成客户友好的周度投资策略邮件/消息

📅 每个交易日（周一至周五）
  
  08:30  晨会（Morning Meeting）
         Paperclip Heartbeat → 宏观策略组 + 个股分析组
         ├── ❶ 宏观研究员: 隔夜外盘 + 汇率/商品 + 重大政策
         ├── ❷ 新闻分析师: 24h 内重要公告汇总 + 评级
         ├── ❸ 风控组: 持仓预警检查（涨跌停/超买超卖/偏离度）
         ├── ❹ 技术面分析师: 今日技术面关键价位
         └── ❺ 所长汇总 → 晨报 Markdown → 推送客户
  
  09:15  集合竞价监控（仅 A 股交易日）
         → 异动检测引擎: 竞价量比异常 / 涨跌停挂单量
  
  09:30  开盘
         → 被动监控（不主动分析，节省 API 成本）
  
  12:00  午间评估（Midday Review）
         Paperclip Heartbeat → 个股分析组 + 交易组
         ├── 上午异动回顾: 涨停/跌停/炸板/天地板/天量
         ├── 持仓偏离度: 是否需要下午调仓？
         ├── Bull/Bear 快速辩论: 下午方向判断
         └── 输出: 午评 → 推送客户
  
  15:10  收盘复盘（Closing Review）
         Paperclip Heartbeat → 个股分析组 + 交易组
         ├── ❶ 当日收益归因: 持仓涨跌原因拆解
         ├── ❷ 持仓状态更新: 现价写入 PositionState
         ├── ❸ 技术面分析师: 收盘形态判断
         ├── ❹ 交易组: 明日操作预案（关键价位 / 触发条件）
         └── ❺ 输出: 收盘报告 → 推送客户
  
  15:30  T+N 复盘与学习（Learning Loop）
         Paperclip Heartbeat → 学习发展组
         ├── 查找 T+5 之前的全部决策记录
         ├── 对比预测 vs 实际走势
         ├── ✅ 正确预测 + 累计 3 次成功 → 生成 Skill.md
         ├── ❌ 错误预测 → 记录避坑要点 → MEMORY.md
         ├── ⚠️ 不确定 → 标记待观察 → 延期再评估
         └── 输出: 技能文档更新（静默，不推送客户）
```

### 5.2 非固定流程 — 客户定制需求

```
触发: 客户通过 OpenClaw (微信/Telegram/WhatsApp/CLI) 发送消息

流程:
  ① 投顾服务组接收 → 意图识别 + 客户偏好注入
     ├── "推荐几只消费股"      → intent: weekly_screening (限定消费行业)
     ├── "600519公告怎么看"    → intent: announcement_analysis
     ├── "被套了怎么解套"       → intent: breakeven_recovery
     ├── "大盘还会跌吗"         → intent: market_outlook
     └── "帮我看看我的持仓风险"  → intent: portfolio_risk

  ② 投顾服务组 → 提交给所长 → 创建 Paperclip 任务

  ③ 所长分派任务给对应部门:
     ├── 行业选股 → 行业研究组
     ├── 个股分析 → 个股分析组
     ├── 大盘研判 → 宏观策略组
     └── 风控评估 → 风控合规组

  ④ 对应部门执行 → 内部讨论 → 输出结果

  ⑤ 投顾服务组接收结果 → 转化为客户友好语言 → OpenClaw → 推送客户

  ⑥ Paperclip 记录完整审计日志: 谁提出 → 谁分析 → 花了多少钱 → 什么结论

  ⑦ T+N 天后该决策进入学习循环
```

## 6. 功能需求

### 6.1 组织管理模块（Paperclip 承载）

| 需求 ID | 需求描述 | 优先级 |
|---|---|---|
| **ORG-01** | 支持创建/编辑公司组织架构（研究所 → 部门 → 岗位） | P0 |
| **ORG-02** | 每个 Agent 有角色（头衔）、汇报线、岗位描述、预算上限 | P0 |
| **ORG-03** | 任务可在组织架构中上下流动（所长分派 → 部门 → 返回） | P0 |
| **ORG-04** | Board 审批模式：招聘新 Agent / 策略变更 / 超预算操作需审批 | P1 |
| **ORG-05** | 多公司隔离：一个实例可运行多个完全独立的研究所 | P2 |

### 6.2 定时调度模块（Paperclip Heartbeat 承载）

| 需求 ID | 需求描述 | 优先级 |
|---|---|---|

### 6.10 知识库模块（TradingAgents 核心基础设施）★ 新增

> 知识库是数字金融团队的"集体记忆"——所有获取的信息、分析的过程、决策的结果、复盘的经验，全部沉淀于此，并可被任何 Agent 在任何时刻主动查询。没有知识库，学习循环就是空中楼阁。

#### 6.10.1 现有知识资产盘点

当前项目已有 5 个存储机制，但均为**被动存储**，Agent 无法在分析过程中主动查询：

| 存储层 | 代码位置 | 存储内容 | 格式 | 查询方式 | Agent 能主动查？ |
|---|---|---|---|---|---|
| DataCache | `dataflows/cache.py` | 原始 API 响应（OHLCV/基本面） | CSV/JSON + 内存 TTL | 按路径直接读 | ❌ 内部用 |
| AnalysisArchive | `analysis_archive.py` | 每次分析结果（ticker/date/结论） | JSON + 年月日 index | 按 ticker/date 过滤 | ⚠️ 仅 ContextAssembler 注入 |
| TradingMemoryLog | `agents/utils/memory.py` | 交易决策+实际结果+反思 | Markdown (Append-only) | 线性扫描 | ⚠️ 仅 ContextAssembler 注入 |
| WikiGenerator | `knowledge/wiki_generator.py` | Agent 可爬取的导航页面 | Markdown 索引 | 文件读取 | ⚠️ 需 Agent 主动读文件 |
| MCP Server | `knowledge/mcp_server.py` | 外部 JSON-RPC 接口 | stdio 协议 | 6 个查询 tool | ✅ 外部可调用 |

**关键缺失清单**：
- ❌ 无统一查询接口 — Agent 不能在同一个调用里查"历史分析+决策结果+市场规律"
- ❌ 无语义/向量检索 — 只能按 ticker/date 精确匹配，查不到"类似当前情况的历史"
- ❌ 无结构化执行追踪 — 只有 Markdown 日志，无机器可读的 Agent 决策链路
- ❌ 无分析知识图谱 — ticker↔industry↔macro↔decision↔outcome 的关系未建模
- ❌ 无技能文档库 — GEPA 学习循环的输出目标不存在
- ❌ Agent 不能"主动查资料" — 只能被动接收 ContextAssembler 注入内容

#### 6.10.3 统一查询接口

Agent 分析过程中通过统一的 `KnowledgeBase` 对象查询，不需要知道数据存在哪一层：

```
# 统一查询接口概念

kb = KnowledgeBase(config)

# 查询 1: 精确匹配 → Layer 1 索引（毫秒级）
kb.query("600519 2026年5月 分析结果", mode="exact")

# 查询 2: 语义相似 → Layer 2 向量（毫秒～秒级）
kb.query("当前白酒板块暴跌，历史上类似情况如何？", mode="semantic")

# 查询 3: 关系推理 → Layer 4 知识图谱（毫秒级）
kb.query("降准对哪些行业影响最大？", mode="graph")

# 查询 4: 混合 → 自动融合多层结果排序
kb.query("我的持仓在当前环境风险多大？类似胜率多少？", mode="hybrid")

# 查询 5: 技能检索 → Layer 5（毫秒级，默认仅摘要）
kb.query_skills(intent="breakeven_recovery", ticker="600519")
# → 返回最匹配技能的摘要; Agent 确认后再加载正文
```

### 7.2 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| **内层 Agent 编排** | LangGraph（Python） | TradingAgents 已有，动态管道扩展成本最低 |
| **外层任务编排** | Paperclip（Node.js） | 原生支持组织架构、Heartbeat、预算、审计 |
| **客户消息通道** | OpenClaw（Node.js） | 25+ 平台覆盖，无需自建 |
| **知识库存储** | 文件系统 + SQLite-vec + JSON | 零外部服务依赖；嵌入式向量搜索；Agent 可主动查询 |
| **学习循环** | GEPA 方法论 + 自实现 | Hermes 的 GEPA 是方法论（MIT 许可），可在 TradingAgents 内按需求定制实现 |
| **数据存储** | 分层：PostgreSQL（Paperclip）+ 文件系统（TradingAgents） | Paperclip 的结构化治理数据需要 ACID；TradingAgents 的分析结果适合 Markdown + JSON 文件 |
| **Agent 间通信** | Paperclip Issues + WebSocket 事件 | 统一任务接口，避免点对点耦合 |

### 8.1 客户请求完整链路

```

  │
  ▼
[OpenClaw] 微信消息 → Gateway → 路由到投顾 Agent 会话
  │
  ▼
[Paperclip] 投顾 Agent 接收 → 创建 Issue "PAP-142: 600519 保本方案"
  │  ├── parent: 客户请求
  │  ├── assignee: 个股分析组
  │  ├── budget: $2.50
  │  └── context: {ticker: "600519", intent: "breakeven", cost: 1850}
  │
  ▼
[TradingAgents] Paperclip Adapter 通过 CLI 触发:
  │  $ tradingagents route --request "客户持有600519，成本1850，现价1758，P&L -4.97%，寻求保本方案"
  │
  ├── Intent Router → 识别: breakeven_recovery
  ├── Context Injector → 注入: PositionState(cost=1850, qty=100), MarketContext(day)
  ├── BreakevenPipeline 执行:
  │   ├── PositionAnalyzer: 成本1850, 现价1758, 亏损4.97%, 持有57天
  │   ├── ScenarioSimulator:
  │   │   ├── 方案A(持有等反弹): 需要涨5.23%回本, 概率评估基于BullBear辩论
  │   │   ├── 方案B(止损离场): 立即亏损-4.97%, 机会成本分析
  │   │   ├── 方案C(分批加仓): 加仓后成本降低, 但风险加大
  │   │   └── 方案D(对冲保护): 买入看跌期权/卖出认购期权
  │   ├── BullBear Debate: "贵州茅台能否在30天内涨回1850？证据是？"
  │   └── PM 整合: 最优方案 + 置信度 + 风险提示
  │
  ▼
[Paperclip] 记录结果: Issue PAP-142 → completed, cost=$1.85, 审计日志归档
  │
  ▼
[OpenClaw] 投顾 Agent 将结果转化为客户语言 → 微信推送
  
  "您的贵州茅台持仓分析结果:
  成本1850元，现价1758元，亏损4.97%

  详细分析报告见附件 ↘️"
```

## 10. 风险与约束

### 10.1 技术风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| **Paperclip 过于年轻**（2026.3 创建，2 个月） | API 变更、Bug 多、不稳定 | 仅使用核心功能（任务+Heartbeat+预算），避免深度定制 Paperclip 源码 |
| **TradingAgents 改造量大** | 开发周期长 | 优先实现 Intent Router + 2 个管道（Standard + Breakeven），其余管道后续迭代 |
| **多平台集成复杂** | OpenClaw 可能有 CVE 风险 | 仅使用消息收发功能，不启用 ClawHub 技能市场 |
| **GEPA 技能质量不可控** | 生成低质量或错误技能 | 技能质量门禁（size 限制 + 语义检查 + 可选人工审核） |
| **LLM API 成本** | 每日多次分析累计费用高 | Paperclip 预算硬停止 + GEPA 缓存减少重复分析 + 逐步迁移到本地 Ollama |

### 10.2 业务风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| **分析结论不构成投资建议** | 合规风险 | 所有输出标注 "本分析由 AI 自动生成，不构成投资建议" |
| **客户过度依赖 AI 决策** | 亏损追责 | Board 审批模式：重大决策需人类确认后再执行 |
| **数据源中断**（如 AkShare 不可用） | 无法分析 | 多数据源 fallback（国信 → Yahoo → Alpha Vantage） |

### 10.3 约束条件

| 约束 | 说明 |
|---|---|
| **交易日历感知** | 所有定时任务必须感知 A 股交易日历，非交易日自动跳过 |
| **T+1 交割规则** | 风控 Agent 必须考虑 A 股当日买入不可卖出的限制 |
| **涨跌停限制** | 异动检测和分析输出必须考虑 ±10%（主板）/±20%（科创创业）限制 |
| **中文输出** | 客户交互层（投顾 Agent → 客户）强制中文；Agent 内部辩论保持英文（推理质量更高） |
