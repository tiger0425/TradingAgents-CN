# 工业级质量差距全面诊断 (Industrial-Grade Quality Gap Analysis)

> 版本：v0.2.14-cn | 日期：2026-06-03 | 范围：LLM 幻觉、工具调用不一致、报告随机变化
> 关联文档：[tool-loop-prevention-system.md](tool-loop-prevention-system.md)、[架构缺陷修复方案.md](架构缺陷修复方案.md)

---

## 一、问题现象

在 v0.2.11-cn 之前的多个生产分析运行中观察到以下症状：

| 症状 | 表现 | 用户影响 |
|------|------|---------|
| 报告内容随机变化 | 同一标的（如 600418 江淮汽车）多次运行，报告结论、工具调用顺序、段落结构均不同 | 无法做 A/B 测试、无法复盘、无法对比 |
| 工具调用不一致 | 同一分析师在相同输入下，有时调用 2 个工具，有时调用 4 个 | 成本不可预测、运行时间差异巨大 |
| LLM 幻觉 | 出现"EPA 2027"、"Class 8"、"ACT Research" 等英文北美卡车行业术语，与 A 股标的完全无关 | 报告质量不可信 |
| 报告格式不统一 | 各分析师输出格式各异，下游解析经常失败 | 自动化流水线断点 |
| 工具被静默过滤 | LLM 收到 `[已过滤不可用工具: get_balance_sheet, ...]` 反馈但不知原因 | LLM 困惑、retry 死循环 |

## 二、根因分析（5 大问题 + 5 大缺口）

### 2.1 根因 #1：LLM Temperature 永远不设 0

**严重度**：🔴 极高 — 这是所有跨 run 差异的最根本来源

| 文件 | 行 | 现状 |
|------|-----|------|
| `tradingagents/llm_clients/openai_client.py` | 243-283 | `get_llm()` 不传 `temperature` 参数 |
| `tradingagents/llm_clients/base_client.py` | 全文 | 抽象基类无 temperature 默认值 |
| `tradingagents/llm_clients/factory.py` | 15-63 | `create_llm_client` 工厂不处理 temperature |
| `tradingagents/bootstrap.py` | 91-150 | `_create_llms()` 不注入 temperature |
| `tradingagents/default_config.py` | 全文 | **整个文件无 `llm_temperature` key** |

> **二次评审修正（v0.2.14-cn）**：`grep -r temperature` 全代码库 0 匹配 — v0.2.13-cn 描述准确。但默认值需更精确：实际是 **`ChatOpenAI` 默认 `temperature=1.0`**（LangChain 默认值），而非 0.7。**0.7 是 OpenAI API 默认值，但 LangChain `ChatOpenAI` 显式覆写为 1.0**。这对非确定性的影响比文档初版更严重——温度越高，输出越随机。

**机制**：每次 LLM 调用 — 4 分析师 + 2 辩论者 (bull/bear) + 1 研究主管 + 1 交易员 + 3 风控辩论者 + 1 PM = **14 次 LLM 调用** — 全部以 LangChain 默认 `temperature=1.0` 运行，**最大随机性**。

**直接影响**：
- LLM 决定调用哪些工具是非确定的 → `bind_tools` 提供的工具集中，LLM 每次选不同子集
- LLM 输出的工具参数是非确定的（如 `get_indicators("macd")` vs `get_indicators("close_50_sma,macd")`）
- LLM 报告内容是非确定的（同样输入、不同输出）
- FIX-8 工具循环检测因 LLM 行为不同而走不同分支

> **新增发现（v0.2.14-cn）—— `get_indicators` 有效指标名定义在 4 个分散位置**

| 位置 | 内容 | 数量 |
|---|---|:---:|
| `market_analyst.py:46-54` system prompt | 12 个指标名明文列举（`close_50_sma, close_200_sma, close_10_ema, macd, macds, macdh, rsi, boll, boll_ub, boll_lb, atr, vwma`） | 12 |
| `technical_indicators_tools.py:6-32` tool schema description | `"e.g. 'rsi', 'macd'"` —— 仅示例，无完整枚举 | 2 |
| `akshare.py:_INDICATOR_DESCRIPTIONS` | `close_50_sma, close_200_sma, close_10_ema, macd, macds, macdh, rsi, boll, boll_ub, boll_lb, atr, vwma, mfi` | 13 |
| `a_stock_data.py:409-460` col_map | `rsi, rsi_14, macd, macds, macdh, kdj_k, kdj_d, kdj_j, boll, boll_ub, boll_lb, sma_5, sma_10, sma_20, sma_30, close_50_sma, close_200_sma, close_10_ema, atr, vwma, mfi` | 22 |

**维护风险量化**：改一个指标名（如 `boll_ub` → `bb_upper`）需要同步 4 个位置。漏一个就导致：
- 行为分裂（不同 vendor 接受/拒绝不同名字）
- 文档 vs 实现脱节
- LLM 在不同路径看到不同"合法"指标名

**修复建议**：
- 建立**单一真理源**（如 `agents/utils/indicator_registry.py`）作为常量字典
- 4 个位置改为从注册表导入
- 新增指标时只改一处

### 2.2 根因 #2：bind_tools 与 ToolNode 不一致（方向：ToolNode 多余）

**严重度**：🟡 中 — **不导致崩溃，但导致功能缺失**

> **二次评审修正（v0.2.14-cn）**：原 v0.2.13-cn 描述"bind_tools 比 ToolNode 少"，但代码验证显示**方向相反**——ToolNode 多于 bind_tools（冗余注册）。这意味着：
> - LLM 永远不会调用到这些"多余"工具 → **不会导致工具调用崩溃**
> - `filter_valid_tool_calls` 会过滤掉 LLM 幻觉的工具调用 → 有安全网
> - 真正的问题是：**功能缺失**（fundamentals analyst 本可以用 `get_balance_sheet` 等细分工具但 LLM 不知道它们存在）

| 分析师 | `bind_tools` 绑定（analyst 文件） | `ToolNode` 注册（bootstrap.py） | 方向 | 影响 |
|---|---|---|---|---|
| market | 4 个 | 4 个 | ✅ 一致 | 正常 |
| **fundamentals** | 1 个（`get_fundamentals`）— `fundamentals_analyst.py:32-34` | **6 个**（+get_balance_sheet +get_cashflow +get_income_statement +get_margin_trading +get_institutional_holdings）— `bootstrap.py:193-200` | ❌ ToolNode 多 5 个 | LLM 不知道 5 个细分工具可用 |
| news | 2 个（`get_news`, `get_global_news`）— `news_analyst.py:29-32` | 3 个（+`get_insider_transactions`）— `bootstrap.py:188-191` | ❌ ToolNode 多 1 个 | LLM 不知道 insider 工具可用 |
| social | 4 个 | 4 个 | ✅ 一致 | 正常 |

**机制澄清**：
- `filter_valid_tool_calls`（`agent_utils.py:303-331`）只放行 `bind_tools` 列表里的工具名
- ToolNode 工具未在 `bind_tools` 中 → LLM 即使想调用也调用不到
- 但 LLM 也不知道这些工具存在（system prompt 未提及）→ 不会主动尝试 → 不会触发过滤
- 所以"已过滤不可用工具"反馈**不来自根因 #2**

**"已过滤不可用工具"反馈的真实根因**（二次评审修正）：
- 根因 #4（双重工具来源）：system prompt 明文列了 12+ 指标名 → LLM 试图以指标名当工具名调用 → 工具名级别幻觉
- 根因 #1（temperature=1.0）：LLM 行为不稳定 → 随机生成工具名 → 过滤

**功能缺失的隐性影响**：
- fundamentals analyst 绑定 1 个工具（`get_fundamentals`），返回综合数据（行情 + 三表）
- LLM 想"细致分析"某张表时，**不知道**可以调用 `get_balance_sheet` 单独查
- 必须通过 `get_fundamentals` 一次性拉全三表 → 单次 token 消耗大、解析复杂、缓存粒度粗
- 这不是"崩溃"，是"功能封装粒度不合理"导致的**调用效率和质量妥协**

**历史背景**：FIX-11 v0.2.11-cn "工具循环四层防护" Layer 1 主动移除了 5 个工具（`fundamentals_analyst.py:32-34`），目的是让 LLM 只调一个聚合工具 `get_fundamentals` 避免循环。但 `bootstrap.py:193-200` 的 ToolNode 注册表未同步更新，留下冗余。

**新增发现（v0.2.14-cn）**：`dynamic_graph_builder.py:31` 中 `"macro_analyst": "market"` 是**死代码映射**——`macro_analyst` agent 不存在但仍被注册，会在动态图构建时静默指向 market_analyst。建议删除或注释掉此映射。

**修复方向（v0.2.14-cn 修正）**：
- 方案 A（推荐）：**对齐 ToolNode 与 bind_tools**——删除 `bootstrap.py:193-200` 中 5 个冗余工具（最保守，无破坏性）
- 方案 B：**对齐 bind_tools 与 ToolNode**——在 `fundamentals_analyst.py:32-34` 恢复 5 个工具（增加 LLM 自由度，但需配合 FIX-11 Layer 2-4 防护）
- **不推荐**：维持现状（功能缺失 + 死代码）

### 2.3 根因 #3：5/12 agent 完全无防幻觉约束，7/12 仅弱约束

**严重度**：🔴 高 — 直接导致英文术语幻觉（EPA 2027 / Class 8 / ACT Research）

| agent | 数量 | 防幻觉约束 | 位置 |
|---|---|---|---|
| 4 分析师 | 4 | ⚠️ `get_degradation_instruction()` 一句中文 "不得编造数据"（`agent_utils.py:65`） | 仅 1 句；社会媒体分析师另有 `Do NOT fabricate sentiment data` |
| 2 辩论 (bull/bear) | 2 | ❌ 无（有意设计：`agent_utils.py:57` "Internal debate agents skip this to keep their prompts lean"） | `bull_researcher.py:51-86`, `bear_researcher.py:51-78` |
| 1 研究主管 | 1 | ⚠️ `get_degradation_instruction()` + `with_structured_output(ResearchPlan)` | `research_manager.py` |
| 1 交易员 | 1 | ⚠️ `get_degradation_instruction()` + `with_structured_output(TraderProposal)` | `trader.py` |
| 3 风控辩论 | 3 | ❌ 无（同上，有意设计） | `aggressive/conservative/neutral_debator.py` |
| 1 PM | 1 | ⚠️ `get_degradation_instruction()` + `with_structured_output(PortfolioDecision)` | `portfolio_manager.py` |

> **统计修正**（v0.2.12-cn）：文档初版 v0.2.11-cn 误将 RM/Trader/PM 归入"无约束"（仅提及其结构化输出），实际这 3 个 agent 同时调用了 `get_degradation_instruction()`。正确统计：**5/12 完全无约束**（2 辩论 + 3 风控），**7/12 有弱约束**（4 分析师 + RM + Trader + PM）。

> **二次评审深化（v0.2.14-cn）—— English 模式下防幻觉约束完全退化**

代码验证发现 **`get_degradation_instruction()` 是语言相关的**：

```python
# agent_utils.py:52-67（简化）
def get_degradation_instruction() -> str:
    if output_language == "Chinese":
        return "降级策略：... 不得编造数据或虚构未获取到的信息 ..."
    else:
        return ""  # ← English 模式下返回空字符串
```

**后果**：
- 当用户配置 `output_language="English"` 时，**4 分析师 + RM + Trader + PM 共 7 个 agent 的防幻觉约束完全消失**
- 这 7 个 agent 的 prompt 中防幻觉部分变成空字符串，等同于 0/12 全部无防护
- English 模式实际是**最脆弱**的运行模式——而国际投资者更倾向用 English，导致国际化场景下幻觉率反而最高

**修复方向**（v0.2.14-cn 新增）：
- `get_degradation_instruction()` 必须为所有语言返回防幻觉约束（中英双语版本都要有）
- 当前代码为 Chinese 模式专属定制是设计错误——防幻觉是普适的，不应该按语言分支

**问题**：
- **fundamentals** 和 **news** 分析师的 prompt 写 "Make sure to include as much detail as possible"（`fundamentals_analyst.py:37`, `news_analyst.py:35`）— **鼓励了幻觉而非抑制**
- market 分析师写 "Write a very detailed and nuanced report"（`market_analyst.py:54`）— 同类问题
- 没有任何 "ONLY use the following tools"、"If data is missing, state clearly"、"Base every claim on specific tool output" 式硬约束
- DeepSeek V4 在缺乏硬约束时倾向"自由发挥"——补全空数据时使用其英文训练语料
- 辩论/风控 agent 缺乏约束是**有意设计**（`agent_utils.py:57`），但此设计选择值得质疑：辩论 agent 同样可能产生幻觉

> **ADR 备注（v0.2.12-cn 新增）— 辩论 agent 跳过防幻觉的设计决策存疑**
>
> **决策来源**：`agent_utils.py:57` 注释 "Internal debate agents skip this to keep their prompts lean"
>
> **当时的设计意图**：辩论 agent 只在已读 analyst 报告基础上做正反方论证，认为 prompt 越短推理越聚焦
>
> **该决策的潜在问题**：
> 1. **辩论 agent 实际并不"lean"** — bull/bear 的 system prompt 已经 50+ 行（`bull_researcher.py:51-86`），加 10 行防幻觉不会显著增加 token
> 2. **辩论 agent 同样可能产生幻觉** — 即使只读 state，LLM 仍可能：
>    - 曲解 analyst 报告中的数据（"基本面分析师说 PE 30，我认为偏低 → PE 应该是 25"）
>    - 引入未在 state 中出现的"常识"（"作为对比，特斯拉 2024 年 PE 是 80"）
>    - 编造行业 benchmark（"行业平均 PE 应该是 50"）
> 3. **辩论 agent 的输出流入 Research Manager / Trader / PM** — 幻觉会向上传播
> 4. **风险辩论 agent 同理** — 三方风控辩论完全无约束，错误风险评估直接进入 PM 决策
>
> **建议（v0.2.12-cn 后重新评估）**：
> - **选项 A**：维持现状，但**监控辩论 agent 的"无来源数字"出现率**（量化后决定是否补救）
> - **选项 B**：在辩论 agent 中加轻量防幻觉（仅"不要发明新数字，只引用报告"），保留 lean 设计
> - **选项 C**：废弃 `agent_utils.py:57` 的特殊跳过逻辑，让辩论/风控 agent 也接受完整防幻觉
>
> **推荐**：**选项 B**（折中），轻量约束 + 量化监控。完全修复成本最低、效果最稳。

> **辩论 agent 的结构性软约束（v0.2.12-cn 新增澄清）**
>
> 表面看辩论/风控 agent 完全无防幻觉，但**它们在结构上受 state 数据流约束**：
>
> | agent | 输入（仅读 state） | 工具绑定 | 结构性约束 |
> |---|---|---|---|
> | **bull_researcher** | `state["market_report"]`, `state["sentiment_report"]`, `state["news_report"]`, `state["fundamentals_report"]`, `investment_debate_state` 历史 | 无（仅文本生成） | **不能凭空写数字**（state 中无任何工具调用），必须基于 4 份 analyst 报告做论证 |
> | **bear_researcher** | 同上 | 无 | 同上 |
> | **aggressive_debator** | `risk_debate_state` + `state["trader_investment_plan"]` | 无 | 不能凭空写数字（state 中无工具调用） |
> | **conservative_debator** | 同上 | 无 | 同上 |
> | **neutral_debator** | 同上 | 无 | 同上 |
>
> **实际幻觉风险重新评估**：
>
> | 幻觉类型 | 辩论/风控 agent 风险 | 原因 |
> |---|---|---|
> | **行业术语错配**（EPA 2027/Class 8） | **低** | analyst 报告已通过 IndustryVerifier 验证；辩论 agent 引用报告内容 |
> | **财务指标编造**（虚构 PE 数字） | **中-高** | 辩论 agent 可"创造"PE 数字来支持论点（如"PE 应是 25 不是 30，因为..."） |
> | **行业 benchmark 编造**（"行业平均 PE 50"） | **高** | 分析师报告中**通常不包含**行业 benchmark，辩论 agent 可凭空写 |
> | **跨标的混淆**（拿 600519 数据说 600418） | **中** | analyst 报告已限定 ticker，但辩论 agent 可能在 cross-ticker 类比时混淆 |
> | **工具幻觉** | **0%** | 辩论/风控 agent 无 `bind_tools`，LLM 不会调用任何工具 |
>
> **结论**：
> - 辩论/风控 agent 的真实风险集中在 **#2（财务指标编造）** 和 **#3（行业 benchmark 编造）**
> - 风险 #2 较隐蔽：辩论 agent 不会写出"PE 是 30"这样的硬数字，但可能写"PE 应该比分析师给的更低，因为..." —— 模糊但有引导性
> - 风险 #3 最严重：分析师只做个股分析，不做行业 benchmark，辩论 agent 完全无源
> - 整体幻觉风险从"5/12 缺防护"降级为"5/12 部分场景需补"
> - **轻量防幻觉约束（ADR 选项 B）可针对性解决**

### 2.4 根因 #4：System Prompt 双重工具来源冲突

**严重度**：🔴 中-高

| 工具来源 | 内容 |
|---|---|
| **`bind_tools()`** | LangChain 自动生成的 tool schema（带 description、parameters） |

**问题**：
- DeepSeek V4 倾向信任 system prompt 字面内容，可能把 `"macd"` 当作工具名直接调用（幻觉工具）
- 或在 `get_indicators(indicator="...")` 的参数中用错值
- `filter_valid_tool_calls` 只能在工具名级别事后过滤，无法在 prompt 层面预防

### 2.5 根因 #5：`_*_data_fully_fetched()` 方法跨 analyst scope 扫描泄露

**严重度**：🟡 中 — 导致运行轮数不一致

> **归因修正**（v0.2.12-cn）：文档初版 v0.2.11-cn 将此问题归因于 `_detect_tool_loop` 的"路径漂移"。经代码验证，`_detect_tool_loop()` 正确使用了 `{analyst_type}_start_idx` 做作用域隔离，实现本身没有问题。真正 bug 在 `_*_data_fully_fetched()` 方法扫描全部 messages 而非 scoped messages（`messages[start_idx:]`）。

| 文件 | 行 | 问题 |
|---|---|---|
| `tradingagents/graph/conditional_logic.py` | 40-113 | `_detect_tool_loop` 基于 message history 决策何时终止（✅ 实现正确，使用 `{analyst_type}_start_idx` 做作用域隔离） |
| 同上 | 119-137 | `_market_data_fully_fetched` 扫描**全部** messages（包含前序分析师的工具调用）→ 市场分析师的 tool call 可能影响基本面/新闻分析师的 early break |
| 同上 | 208-221 | `_fundamentals_already_fetched` 同样跨分析师扫描全部 messages |
| 同上 | 282-304 | `_inject_break_message` 注入的 break message 因 LLM 调用顺序不同而不同 |
| 同上 | 341-388 | `should_continue_debate` 用 `DebateQualityTracker` 评分 < 0.4 + 收敛检测提前终止 |

**问题**：
- 因 LLM 决定调用哪些工具是非确定的（根因 #1），`_*_data_fully_fetched` 的真假值因 run 而异
- `should_continue_*` 走不同分支 → 同一输入下分析师可能跑 2 轮 vs 4 轮、辩论可能 4 轮 vs 5 轮
- `_market_data_fully_fetched` 跨分析师扫描，fundamentals 看到 market 的工具调用记录时可能误判"已获取"（不同 analyst 的 tool 集合通常不重叠，实际触发概率低但理论上存在）

**位置**：`default_config.py` 全文

### 2.8 架构缺口 #3：通用内容验证层缺失

**位置**：所有 agent 返回 `{"messages": [result], "report": content}` 后

**已有防护（非空白）**：
- `IndustryVerifier.verify_industry_consistency()`（`verifier.py`）：两层验证（规则层关键词匹配 + LLM 语义回退），检测报告是否使用了不适用的行业术语/指标
- `DebateQualityTracker`（`debate_quality.py`）：辩论质量评估（新颖度/冗余/收敛检测），评分 < 0.4 时提前终止辩论

**已有防护覆盖率矩阵**（基于代码追踪 + 经验估计，无实际跑分数据）：

| 幻觉类型 | 现有防护 | 实际覆盖率（估计） | 主要盲区 |
|---|---|:---:|---|
| **行业术语错配**（EPA 2027 / Class 8 / ACT Research） | IndustryVerifier 规则层 | **~90%** | 低频未收录的英文术语 |
| **指标名错配**（用 SaaS 指标分析卡车公司） | IndustryVerifier 规则层 + IndustryClassifier 注入 | **~85%** | 跨类型指标（"日活" 用于 B2B） |
| **财务指标编造**（虚构 PE/PB/营收数字） | ❌ 无 | **0%** | LLM 可凭空写出"PE 30.5 倍"无验证 |
| **工具未调用就出数据** | filter_valid_tool_calls 间接防御（被过滤但可能继续写） | **~30%** | LLM 把过滤反馈吞了继续编 |
| **格式/结构不一致**（缺表格、缺来源） | system_message 建议（"Make sure to append a Markdown table"） | **~40%** | 仅口头建议，无强制 |
| **跨报告互相矛盾**（market 说涨、fundamentals 说跌） | ❌ 无 | **0%** | 无 cross-validation 阶段 |

**关键发现**：
- 行业类幻觉**已被较好覆盖**（90%/85%），这是 v0.2.12-cn 的关键保护
- **真正的真空区是"非行业类数据编造"** — 财务数字、PE/PB 等可能完全虚构
- 缺口比初版估算的窄 — 从"完全无防护"降级为"局部盲区"

**缺失内容**：
- 无 "报告必须包含至少 X 个引用" 验证
- 无 "报告不能包含 hallucinated tickers" 验证
- 无 "non-empty content" 检查（`analysts/*:82-89` 用 ToolMessage 替代空内容是**隐式掩盖失败**）
- 无格式合规性检查
- 无跨报告一致性校验
- **无"数字必须出现在某次工具输出中"验证** — 财务编造的核心盲区

**后果**：
- LLM 失败被静默接受（用旧 ToolMessage 兜底）
- 输出质量无 SLA 保障
- 无法构建 LLM-as-judge 评估闭环
- 行业验证虽然捕获了部分幻觉（如 EPA 2027 / Class 8），但无法捕获行业无关的数据编造

### 2.9 架构缺口 #4：主消息历史 context window 管理缺失（已存在但覆盖范围不足）

**位置**：
- `tradingagents/agents/utils/agent_utils.py:196-300` `sanitize_messages_for_deepseek` — 只保留最后一个工具周期
- `tradingagents/graph/context_manager.py:42` `DEBATE_TOKEN_BUDGET = 4000` — 仅限辩论注入
- `tradingagents/default_config.py` — 无 `max_context_tokens`

**已有防护（非空白）**：
- `ContextWindowManager`（`context_manager.py`）：提供辩论场景的三级压缩策略（预算监控 → LLM 摘要 → 硬截断回退），`REPORT_TOKEN_BUDGET = 8000`、`knowledge_token_budget = 25000`
- `summarize_if_needed()`：辩论注入文本超预算时自动压缩

> **二次评审深化（v0.2.14-cn）—— ContextWindowManager 覆盖范围分解**

| 数据流 | 是否有 budget 控制 | 是否有 LLM 摘要 | 是否有硬截断回退 |
|---|:---:|:---:|:---:|
| **辩论历史注入**（bull/bear/risk_debate 看到的 `debate_history`） | ✅ `DEBATE_TOKEN_BUDGET=4000` | ✅ `_llm_summarize()` | ✅ 硬截断回退 |
| **知识库注入**（所有 analyst 看到的 `knowledge_context`） | ✅ `knowledge_token_budget=25000` | ⚠️ 仅 cache/lessons 移除，不摘要 | ✅ 优先级移除 |
| **报告注入**（辩论 agent 看到的 4 份 analyst 报告） | ✅ `REPORT_TOKEN_BUDGET=8000` | ✅ `_format_reports_summary()` | ✅ 硬截断 |
| **主消息历史**（`state["messages"]` 自己的 tool 循环） | ❌ 无 | ❌ 无 | ⚠️ `sanitize_messages_for_deepseek` 按"最后一个工具周期"截断（非 token 计数） |
| **最终报告组装**（`executor.py:_extract_report` 4 报告 + 计划） | ❌ 无 | ❌ 无 | ❌ 无 — 报告长度可无限增长 |

**关键发现**：
- `ContextWindowManager` 设计良好但**只服务于"注入"层**（prompt 中的额外内容）
- **不覆盖"主消息历史"层**（`state["messages"]` 自身的 tool 循环）
- 缺口不是"完全没管理"，而是"**前 80% 已覆盖，最后 20% 真空**"——文档之前对缺口的描述过于悲观

**缺失内容**：
- 主消息历史（`state["messages"]`）随 tool 调用不断增长，无 token 计数控制
- 多轮辩论（≥3 轮 × 2 边 + 风控 3 边）可能导致 `messages[]` 超过模型 context window（DeepSeek V4 默认 32K）
- 分析师 prompt 无 token 预算控制
- `ContextWindowManager` 只覆盖辩论历史注入和知识注入，不覆盖 agent 自己的 `messages` history

**真实风险场景**：
1. 某个 analyst 反复进入工具循环（FIX-8 防护下不会超过 Layer 3 上限，但 messages 仍持续增长）
2. 跨多轮辩论（bull/bear × N 轮 + risk × N 轮）`messages[]` 累计可达 50-80 条
3. 单次 LLM 调用的 system prompt + messages 超过 32K → API 400 错误
4. Graph 崩溃，需要从头重跑

**后果**：
- 风险：context window 溢出 → API 400 错误 → 整个 graph 崩溃

### 2.10 架构缺口 #5：build_instrument_context 静默 LLM 调用

**位置**：`tradingagents/agents/utils/agent_utils.py:130-134`

**机制**：
```python
framework = IndustryFramework().lookup(industry, quick_llm=quick_llm)
```

当 industry 不在本地框架数据库中（例如冷门行业）时，调用 `IndustryFramework().lookup()` 触发 `_auto_generate()` → LLM 调用。

**问题**：
1. 使用**同一个 `quick_llm`** 作为分析师后续调用
2. 在提示词构建阶段**静默发生**，无日志
3. 结果被塞进 system prompt
4. 如果 auto-generation 产生幻觉（错误的 anti_patterns 或 correct_metrics），整个分析的基础都被污染
5. 消耗未计入的 token 和延迟

### 5.3 P1-1：创建全局 `ANTI_HALLUCINATION_INSTRUCTION`

**新文件**：`tradingagents/agents/utils/prompt_constants.py`
```python
"""全局 prompt 常量 — 防幻觉、结构化输出约束"""
from tradingagents.dataflows.config import get_config

def get_anti_hallucination_instruction() -> str:
    """全局防幻觉指令，注入所有 12 个 agent 的 system_message。
    
    关键约束（按重要性排序）：
    1. 只使用本 prompt 列出的工具 — bind_tools 自动注入工具 schema
    2. 数据缺失必须明确标注 — "数据不可用"，禁止编造
    3. 每个结论必须有工具输出引用 — 引用具体的 tool_message
    4. 不得编造财务指标 — 营收、净利润、PE 等必须来自 get_fundamentals
    5. 不得混入非该标的市场知识 — A 股分析不要引用美股/欧股术语
    """
    cfg = get_config()
    lang = cfg.get("output_language", "Chinese")
    
    if lang == "Chinese":
        return """
**【防幻觉约束 — 必须严格遵守】**

1. **只使用 bind_tools 列出的工具**。禁止调用未列出的工具名。
2. **数据缺失必须明确标注**。如果某项数据未获取到，写"[数据不可用]"而非编造。
3. **每个结论必须有工具输出引用**。在报告末尾的"数据来源"部分列出引用的工具名。
4. **不得编造财务指标**。所有数字必须来自 get_fundamentals、get_stock_data、get_indicators 等工具的真实输出。
5. **A 股分析禁止引用非中国市场的行业术语**。如出现 EPA 2027/Class 8/ACT Research 等美股/欧股术语，立即修正为 A 股对应概念。
6. **报告语言**：整篇报告必须用{language}书写，禁止中英文混用。
"""
    else:
        return """
**【Anti-Hallucination Constraints — MANDATORY】**

1. Use ONLY tools listed in bind_tools. Never invent tool names.
2. If data is missing, state "[Data Unavailable]" — never fabricate.
3. Every claim must reference specific tool output.
4. Never fabricate financial metrics. All numbers must come from tool outputs.
5. Do not inject non-target-market industry knowledge.
6. Report language: write entire report in {language}.
"""
```

## 十一、跨标的报告风格一致性缺口诊断（v0.2.15-cn 补充章节）

> **背景**：v0.2.11 ~ v0.2.14-cn 章节主要分析"同一标的跨 run 不稳定"和"同一 run 内各 analyst 格式不统一"两类问题。**本章补充第三维度——"不同标的（即使同行业/同类型）生成的报告风格不一致"**——这是文档前序章节未独立分析的现象。
>
> **补充原因**：v0.2.15-cn 在用户实际反馈中发现，多标的对比场景（CLI batch 模式、API 多用户并发、Wiki 批量索引）下，"风格不一致"问题比"跨 run 随机"问题更影响工业级可用性。
>
> **本章与前序章节关系**：本章不修改前序任何结论，而是在 v0.2.14-cn 框架基础上**新增** 4 个跨标的特异性根因（命名为"跨 1"~"跨 4"），并给出独立修复方案。

---

### 11.1 现象描述（与前序 5 个症状并列的第 6 类）

| 维度 | 表现 | 影响场景 |
|------|------|---------|
| 报告长度跳跃 | 600519（茅台）报告 800+ 字、宇通客车 600066 报告 200 字（数据缺失无 fallback 骨架） | CLI batch 输出文件大小差异巨大；用户感知混乱 |
| 章节顺序漂移 | 标的 A 报告"市场→基本面→新闻→舆情"；标的 B 报告"基本面→市场→舆情→新闻" | 自动化 Markdown 解析器对章节序号失效 |
| 段落结构漂移 | 同一分析师对两个标的写出完全不同的段落组织（如"PE 估值在前"vs"营收分析在前"） | 用户无法横向对比 |
| 行业术语密度失衡 | 银行标的报告出现 12 个金融术语、SaaS 标的报告出现 18 个 SaaS 术语 | 难以快速判定"该报告是不是合理地谈到了行业核心" |
| 缺数据 fallback 格式 | 数据缺失时 LLM 自由发挥 fallback 段落（"由于数据不可用..."） | 风格随机、不可控 |

**与前序症状对比**：
- 前序 5 个症状聚焦于"同一 run 内 / 同一标的跨 run"
- 本章第 6 类症状聚焦于"**不同标的 × 同一 run**"——一个独立维度

---

### 11.2 根因分析（4 个跨标的特异性根因）

#### 跨 1：行业框架注入格式不统一

**严重度**：🟡 中 — 是跨标的 prompt 形状漂移的首要来源

| 位置 | 内容 |
|---|---|
| `tradingagents/agents/utils/agent_utils.py:130-134` | `framework = IndustryFramework().lookup(industry, quick_llm=quick_llm)` |
| `tradingagents/industry/framework.py` | 不同 industry 的 `anti_patterns` 列表长度、措辞、注入位置无统一契约 |

**机制**：
- 银行 industry → 注入 3 条 anti_patterns（共 150 字）→ prompt 较短
- 重卡 industry → 注入 8 条 anti_patterns（共 600 字）+ 行业框架附加段 → prompt 较长
- SaaS industry → 注入 0 条 anti_patterns + 仅有 correct_metrics 列表 → prompt 结构最简

不同长度/格式的注入让 LLM 看到的 prompt "形状"不同 → LLM 倾向于**按 prompt 形状等比输出**（长 prompt → 长报告、含列表 → 报告里也倾向用列表）→ 跨标的输出形状漂移。

**与前序根因的关联**：
- 不是根因 #1（temperature）的二次表现
- 不是缺口 #2（schema 缺失）的同义——schema 缺失是"没形状"，跨 1 是"形状不一致"
- 是**行业框架子系统**自身的契约缺失

---

#### 跨 2：报告组装层无统一 wrapper（`_extract_report` 字符串拼接）

**严重度**：🔴 高 — 是跨标的 section 内部结构漂移的直接来源

**位置**：`tradingagents/graph/executor.py:299-329`

```python
return "

".join(parts) if parts else ""
```

**机制**：
- 标题格式固定 ✅（`--- Market Analyst ---`）
- 但每个 section **内部**结构完全由 LLM 决定 ❌
- 无 "ReportRenderer" 中间件按 Pydantic schema 强制渲染

**与缺口 #2 的关系**：
- 缺口 #2 解决了"agent 输出结构化"（Pydantic 对象）
- 跨 2 解决了"组装层渲染结构化"（统一 wrapper）
- 两者必须**同时落地**，否则只补一边仍会漂移

---

#### 跨 3：行业间合理差异 vs 风格混乱的边界未定义

**严重度**：🟡 中 — 影响"什么该保留、什么该消除"的形式化判断

| 类型 | 例子 | 应保留 / 应消除 |
|---|---|---|
| 行业特异术语 | 银行含"净息差"、重卡含"销量同比"、SaaS 含"ARR" | 应保留 ✅（行业特性） |
| 行业特异指标 | 银行含"不良率/拨备覆盖率"、SaaS 含"NDR/Logo Count" | 应保留 ✅（行业特性） |
| 报告长度 | 茅台 800 字 vs 冷门标的 200 字（无 fallback 骨架） | 应消除 ❌（缺失数据无 fallback） |
| 章节顺序 | 标的 A "市场→基本面" vs 标的 B "基本面→市场" | 应消除 ❌（应统一） |
| 段落结构 | 标的 A "PE 估值在前" vs 标的 B "营收分析在前" | 应消除 ❌（同 analyst 应一致） |

**当前状态**：`docs/` 目录下无任何文档化"行业特异字段白名单"——导致 IndustryClassifier 输出的 industry 分类是**无下游契约的**。

---

#### 跨 4：Planner 模板系统只覆盖意图骨架，不覆盖报告骨架

**严重度**：🟡 中 — 影响模板复用率和报告结构稳定性

**位置**：`tradingagents/planner/llm_planner.py:97-105` 的 `_generation_mode`

```python
plan["_generation_mode"] = "template_exact"  # 或 template_refined / llm_full
```

**机制**：
- `template_exact` / `template_refined` / `llm_full` 三种模式**只决定调哪些 agent（workflow）**
- **不决定**每个 agent 输出什么字段、字段顺序、必选 vs 可选

**实际效果**：
- 即使 `tpl_standard_analysis` 模板复用 100%
- 每次执行时，analyst 输出仍是 LLM 自由发挥
- 模板复用的收益**只限于 agent 数量和顺序**，不延伸至报告结构

**与跨 2 的关系**：
- 跨 2 是"组装层不统一"（下游）
- 跨 4 是"模板层不覆盖报告结构"（上游）
- 两者从不同方向收敛到同一目标：报告结构稳定

---

### 11.3 与前序根因的协同 / 独立分析

| 维度 | 前序根因 | 本章根因 |
|---|---|---|
| 解决"跨 run 不稳定" | 根因 #1（temperature=0）+ 缺口 #4（context window） | — |
| 解决"同一 run 内 analyst 格式不统一" | 缺口 #2（Pydantic schema） | — |
| **解决"跨标的风格不一致"** | ⚠️ 部分（缺口 #2 覆盖 ~40%） | ✅ **本章 4 个根因专攻此维度** |
| 行业相关 | 缺口 #1（行业检测） + 2.10（build_instrument_context 静默调用） | **跨 1**（注入格式不统一） + **跨 3**（字段白名单缺失） |
| 报告组装 | 缺口 #3（验证层） | **跨 2**（wrapper 缺失） |
| 模板复用 | — | **跨 4**（模板只覆盖 workflow） |

**关键结论**：缺口 #2（Pydantic schema）是"必要不充分"条件——只补缺口 #2 仍会留下 ~60% 的跨标的风格漂移。本章 4 个根因是缺口 #2 之外的**必要补充**。

### 12.2 微观缺口 1：temperature 差异化未落地（🔴 代码层面缺陷）

**现象**：`llm_debate_temperature=0.3`、`llm_risk_temperature=0.2`、`llm_decision_temperature=0.1` 三个配置键已在 `default_config.py:29-34` 定义，API 调用链路（bootstrap → client → LLM）也已打通。但 **`bootstrap.py:_create_llms()` 仅返回一对 `(deep_llm, quick_llm)`，未创建针对辩论/风控/决策层的独立 LLM 实例**。

**后果**：
- 辩论 agent（bull/bear）实际使用 `quick_llm` 的 `temperature=0.0`，未获得 0.3 的差异化温度
- 风控 agent 同理
- 文档中"辩论 agent 应有差异化温度以保留观点多样性"的设计意图**未在代码中实现**

**诊断验证**（2026-06-03 端到端实测）：

```python
>>> deep_llm.temperature   # ResilientLLM → None（DeepSeek-Reasoner 不暴露）
>>> quick_llm.temperature  # 0.0 ✅ 正确
```

**修复方向**：
- `_create_llms()` 返回 4 层 `dict` 而非 2 个 LLM —— `{analyst, debate, risk, decision}`
- 各 agent 创建时按角色取对应 LLM

### 12.4 端到端确定性实测（5 次 600418）

**测试条件**：
- LLM provider：`deepseek`，quick_llm=`deepseek-chat`（temperature=0.0），deep_llm=`deepseek-reasoner`
- 同一 Planner plan（仅生成 1 次，5 次执行复用）
- 同一触发词和 Context

**结果**：5 次产生 **5 种不同报告**（MD5 各不相同），报告长度 8,976~15,222（方差 ~70%）。

**根因分解**：

| 因素 | 可控？ | 说明 |
|------|:---:|------|
| 外部数据源实时变化（akshare API） | ❌ 不可控 | `get_stock_data()` 等返回实时行情，每次不同 |
| 辩论轮数浮动（5~8 轮） | ⚠️ 部分可控 | 辩论质量评分 < 0.4 触发提前终止，不同 run 触发时机不同 |
| DeepSeek-Reasoner 不暴露 temperature | ❌ 不可控 | Planner 用推理模型，天生非确定 |
| 风险辩论轮数浮动（5~8 轮） | ⚠️ 部分可控 | 同上 |

**结论**："5 次 MD5 一致"的目标用**真实数据源**时不可达——这是**架构设计选择**（实时数据 + 推理模型 + 辩论），不是代码 bug。文档第六章的"Jaccard ≥ 0.85"目标应修正为**"在 mock 数据源下可验证，实际场景下作为趋势指标"**。

**修正后的验收标准**：

| 指标 | 原始目标 | 修正目标 | 理由 |
|------|---------|---------|------|
| 5 次 MD5 一致 | 100% | 不可达（移除） | 实时数据源天然变化 |
| Jaccard ≥ 0.85 | 期望 | 改为 mock LLM 环境下可测 | 真实场景不可验证 |
| 硬幻觉关键词（AI/科技类） | 0 | **0** ✅ 已验证通过 | IndustryVerifier 有效 |
| 工具调用顺序一致 | ≥ 90% | **已验证**（4 analysts 均全量调用） | bind_tools 对齐修复生效 |
| 中英双语防幻觉 | 均非空 | **✅ 已验证** | 中 354 字符 / 英 423 字符 |
